# GenClaw：代码驱动的智能体图像生成

原论文题目：*GenClaw: Code-Driven Agentic Image Generation*

说明：以下为基于论文源码整理的中文翻译稿。为保证可读性，我对少量 LaTeX 标记、图表引用和重复表述做了轻微整理，但不改变原意。

## 摘要

图像生成模型近年来已从“文本条件下的像素合成”逐步演进为具备视觉理解与工具调用能力的多模态智能体。然而，现有智能体仍严重受制于底层黑盒图像模型。它们的工作流基本被困在“不断改写提示词以改进生成结果”的循环中，却缺乏直接操控画布的机制。归根结底，LLM 作为真正“画笔”的潜力，即用于精确视觉构造的能力，仍然远未被充分释放。

本文提出 **GenClaw**，一种**代码驱动的智能体图像生成范式**，使智能体能够像人类艺术家一样创作：先构思，再起稿，最后上色。具体而言，智能体首先通过搜索与推理构建概念知识和上下文；然后利用代码（如 SVG、HTML、Three.js）渲染可执行的视觉草图；最后借助图像生成模型补充纹理、材质与照片级真实感。在这一工作流中，代码充当连接“语言推理”与“像素合成”的可控中间画布，将程序逻辑与生成模型的视觉表现力无缝结合起来。通过把图像生成从黑盒式范式转变为更接近真实人类创作的分阶段流程，**GenClaw** 为构建高可控、可解释的视觉生成系统迈出了一步。

## 1. 引言

近年来，图像生成取得了显著突破，其底层范式也在持续变化：从早期“文本驱动合成图像”的系统，逐步发展到能同时整合视觉理解与视觉生成的统一架构。早期的 GAN 与扩散模型大幅推动了高质量像素合成的发展，但这些模型本质上主要是“文本到图像翻译器”，在深入理解用户意图和处理复杂逻辑推理方面能力有限。

随着研究推进，统一的理解-生成模型，如 GPT-Image、Qwen-Image 和 Nano-Banana，将图像生成能力提升到新的高度。依赖更大规模的模型能力与训练数据，这类大型多模态模型在世界知识融入、复杂指令遵循、排版文字渲染等高难任务上表现出色，为下一代视觉生成系统奠定了基础。

图像生成的最新进展已经不再局限于“一步到位”的端到端像素合成。生成模型的角色正从“被动像素响应器”向“生成智能体”演化，即能够自主规划、调用工具，并根据反馈持续优化结果。商业系统和学术工作都已开始把搜索、审阅、推理、编辑等能力引入图像生成流程中，推动生成系统向“创意智能体”演化。

但深入分析现有图像生成智能体后，可以发现一个根本限制：虽然智能体在补全上下文与审查结果方面发挥了作用，但最终视觉合成仍几乎完全依赖端到端的文生图黑盒模型。智能体更像是在给印刷机下命令，只能通过不断改写提示词参与一个带随机性的“黑盒抽奖”过程。结果就是，它实质上沦为一个高级提示词优化器。

与之相反，真正的人类艺术创作是一个透明、分阶段的流程：从概念构思、空间规划，到线稿草图，再到上色和细化。当前智能体范式的内部信息流却几乎完全依赖自然语言，而自然语言在表达绝对空间坐标、精确物体数量、复杂排版布局以及层级遮挡关系时，天生存在显著歧义。这导致智能体无法真正获得对视觉空间结构的实质性操作控制。瓶颈的根源很清楚：现有智能体缺乏一支真正适合自身模态专长的“画笔”。

为缓解自然语言在空间表达上的局限，我们需要一种新的“数字画笔”作为中间表示：它应当更可控，并且天然适合 LLM。近年来，视觉代码生成与分层表示逐渐进入研究视野，并成为结构化视觉生成的潜在候选。与黑盒像素合成不同，代码，尤其是 SVG 这类向量程序，天然具备结构显式、逻辑严谨、可编辑、可渲染验证等优点，也与代码智能体擅长的编程与调试能力高度契合。

事实上，前沿大语言模型已经展现出很强的代码绘图与前端渲染潜力，可以像画家打线稿一样，通过代码构建图像骨架。但代码擅长的是“逻辑与结构”，而不是“像素与纹理”。如果完全使用纯代码渲染最终图像，结果通常停留在扁平图标、UI 或其他规则性较强的任务层面，因为纯代码在复杂光照、羽化边缘、头发、自然纹理等高频真实细节的表达上存在明显上限。恰恰相反，逼真图像生成正是图像生成模型擅长的领域。

基于这种天然的能力互补，本文提出了一种新的**代码驱动智能体图像生成范式**，并基于此构建了系统 **GenClaw**。其思想是让图像生成智能体真正模仿画家的创作角色：

- 先通过搜索与推理获取准确实体知识与上下文，完成**概念构思（Conceptualize）**。
- 再把代码写作当作手中的“数字画笔”，在画布上用结构化方式表达视觉意图，完成**草图绘制（Sketch）**。
- 最后让图像生成模型更像“上色师”，在结构化草图上补充高保真纹理、材质与真实感，完成**上色（Color）**。

在多项复杂视觉任务上，这种解耦架构表现出明显潜力，尤其体现在以下方面：

- **构图更可控**：复杂指令可被编译为带有坐标与数量约束的视觉代码，从而缓解传统模型在物体计数与空间关系上的幻觉。
- **文本排版更可靠**：把文字渲染交还给 SVG/HTML 等代码层，可减少拼写混乱，并更细粒度地控制字号、对齐和层级。
- **可辅助物理规律模拟**：对于复杂物理场景，可调用 HTML、Three.js 等先构建 3D 或几何参考，再由模型做视觉化。
- **结构化视觉条件编辑**：可把自然语言转化为结构化视觉代码，更直接地操作底层模型的视觉条件输入。
- **分层图像编辑更灵活**：借助图层与 JSONL 结构，智能体可在局部编辑时精准隔离目标区域，减少对未修改区域的破坏。

因此，真正的范式转变并不只是从简单提示工程升级到更复杂的提示工程，而是从“端到端黑盒生成”迈向“像人类画家一样作画”。GenClaw 的生成流程更贴近真实的人类创作过程，因此在透明性上具有显著优势。生成失败时，我们可以清晰追踪问题究竟出现在搜索得到的上下文、LLM 生成草图代码的逻辑，还是最终从草图到逼真图像的渲染阶段。

进一步地，随着 Claude Code、Codex 等代码智能体展现出惊人的泛化能力，一个自然问题出现了：**代码智能体能否被用于视觉生成？** 过去的图像生成模型主要是被动聊天式系统，而未来显然会转向智能体范式。GenClaw 正是在这一方向上的初步探索。

## 2. 相关工作

### 2.1 图像生成模型

近年来，图像生成已从文本条件像素合成发展到能够原生同时支持视觉理解与生成的统一大型多模态模型。早期扩散系统如 Stable Diffusion、DALL-E 等推动了高质量图像合成的快速进步。如今的生成模型已经能够合成人眼几乎难以分辨真伪的逼真图像。

之后，Janus 等工作开始在单一框架内联合建模视觉理解与生成，标志着研究重点逐步从单功能生成器转向更完整的多模态系统。GPT-4o 进一步推动了这一趋势，其受到关注的不仅是生成质量，还有复杂视觉推理、文字渲染与指令遵循能力。后续模型如 BAGEL、Qwen-Image、Nano-Banana 等则继续增强架构设计、推理注入、复杂排版和高保真编辑能力。

进一步的发展正在把图像生成模型推向智能体化。闭源系统如 Nano-Banana-Pro 与 FLUX 2 Pro 已开始把搜索与审阅模块引入生成闭环。总体来看，图像生成的发展路径正在从单一像素合成器，转向统一的理解-生成模型，再进一步走向智能体式图像生成器。

### 2.2 面向图像生成的智能体

随着大语言模型能力增强，Codex、Claude Code 等代码智能体表明，这类模型正在从对话助手进化为**可执行智能体**：它们能读取状态、调用工具，并根据反馈修正动作。这一趋势也催生了图像生成领域的平行智能体研究。

如 Think-Then-Generate 和 GenAgent 将高层理解与具体生成显式解耦，在合成前插入多模态推理步骤；Mind-Brush 把搜索和推理工具引入开放域创作，用以弥补实时知识缺口；JarvisEvo 与 RefineEdit-Agent 构建了多轮闭环编辑框架；Lovart 和 TapNow 等商业系统也把创意界面从单一提示框转向多工具协作工作流。

与本文最接近的是 CoCo，它使用 Matplotlib 代码生成结构化草图，再将其细化为最终图像，验证了“可执行程序作为中间表示”的可行性。但 CoCo 仍然较强依赖单一统一模型同时完成代码生成与像素细化，因此没有充分利用解耦架构在复杂任务上的优势。

更广泛地看，现有图像生成智能体大多仍是“高级提示词优化器”或“知识检索器”，其内部信息流依旧主要通过自然语言传递。因此，语言模型对视觉空间结构的实际控制力仍然有限。相比之下，本文的代码驱动范式把中间表示具体化为**可执行视觉代码**，使语言模型可以直接参与构图、排版与分层构造，而图像生成模型则专注于最终的纹理表达与照片级渲染。

### 2.3 视觉代码生成与分层表示

受大语言模型在逻辑推理和代码编写方面的天然优势驱动，使用视觉代码与分层表示来引导图像生成，正成为活跃研究方向。与直接在像素空间合成不同，这些方法把视觉内容表示为由路径、图形、文字和层级构成的向量程序（如 SVG、HTML），其特点是可编辑、可无损缩放、结构显式。

OmniSVG 首次把高质量 SVG 生成建模为统一多模态任务；InternSVG 进一步把 SVG 的理解、编辑、生成统一进同一框架。随着基础模型增强，通用语言模型已经展现出零样本代码绘图的可观潜力，例如 Kimi、DeepSeek 等模型可直接通过代码构建复杂物理结构或渲染网页界面。与此同时，VCode、Vec2Pix、Qwen-Image-Layered 等工作也表明，SVG 与分层表示可以成为图像压缩、修订、编辑与高保真像素生成之间的重要桥梁。

但现有纯代码生成研究主要仍局限于图标、UI 布局、独立组件等规则性较强的任务，对于复杂场景整体构图或开放域语义组织的支持仍有限。并且纯代码在光照、头发、自然纹理等高频写实细节上的表达本身有上限。因此，本文不把视觉代码视作最终产品，而把它视作智能体内部的**代码化中间草图**，用于拆解图像、组织布局并支持迭代修订；最终的逼真渲染仍交由图像生成模型完成。

## 3. 方法

### 3.1 整体框架

本文提出一种代码驱动的图像生成智能体框架，其核心思想是：把“向模型直接发送提示词然后生成图像”的黑盒过程，改造成更接近人类绘画方式的分阶段流程。人类创作图像时，通常不会一开始就得到最终图，而是先在脑中形成想法，必要时搜索参考或进行推理；再画草稿，确定对象、位置、文本与主要结构；最后补充纹理、光照与真实感。

在智能体系统中，作者把这一过程分解为三层：

1. **认知结构化层（Cognitive Structuring Layer）**
2. **可执行画布层（Executable Canvas Layer）**
3. **视觉生成与审查层（Visual Generation and Review Layer）**

第一层以 VLM/LLM 为核心认知模块，并结合搜索、知识库与推理工具，在图像生成前主动完成理解工作，包括理解用户意图、理解参考图、补充世界知识，以及完成数学、地理、物理等推理。这些任务更适合由多模态智能体承担，而非交给图像生成模型本身。

第二层中，智能体把第一层组织出的结构化记录转化为一个**可执行画布**，例如 SVG、HTML/CSS、Python 绘图或简单 3D 脚本。此时，代码就是智能体的“数字画笔”：与其让智能体像人类那样在 GUI 里用鼠标一点点画，不如让它通过更适配自身能力的 CLI/代码形式，直接构建对象、文字、坐标、图层和可编辑单元。

第三层中，智能体调用现成的图像生成模型（如 Qwen-Image、Nano Banana）把中间的可执行画布渲染成视觉上丰富的最终图像。随后再由 VLM 自动审查，或由用户交互式审查，确保结果与目标一致。因为整个工作流更透明，用户可以基于中间布局和最终输出进行高度动态、细粒度的调整。

与一步式 prompt-to-image 方法相比，该框架让智能体的“思考”不再只是停留在提示词改写，而是进一步物化为一个可执行的画布状态。最终图像不再完全由图像模型从提示词中“猜出来”，而更像人类白盒创作：先思考和构思，再搭草图，最后完成图像。

### 3.2 认知结构化层

这一层的目标，是把图像生成前的理解与推理任务，从图像生成模型中解耦出来，并在用户意图与可执行画布之间建立明确的认知轨迹。这对应于人类绘画中的思考与构思阶段。

现有生成模型擅长把条件映射为视觉内容，但并不总适合处理复杂意图解析、世界知识检索或符号推理。与直接扩展提示词不同，这一层不尝试直接生成最终图像描述，而是显式解析用户意图，补全缺失知识，进行必要推理分析，并把结果组织成结构化记录。

具体来说，智能体首先执行意图理解。对于普通图像生成任务，可能只需轻量的提示词整理；但当任务涉及具体或动态概念，如长尾实体、实时事件、地理位置、文化符号或专业对象时，模型内部知识往往不足以支持准确生成，这时智能体会调用搜索工具来补足相关事实，填补认知空缺。

对于涉及数学、地理、物理等复杂理解与推理的任务，智能体会先基于 VLM 的推理能力显式得出中间结论，再把隐式关系转换为视觉约束。例如，几何任务中，智能体会先算出数值答案，再把它可视化。这样，图像生成之前的认知工作就被显式化，而不是把所有理解压力都压给最终图像模型。

在完成意图理解、知识补全与推理分析之后，智能体将结果整理成 **JSONL 风格的结构化记录**。这种记录不追求自然语言描述的丰富性，而强调可执行性与可追溯性：它要明确说明应出现哪些对象、应渲染哪些文本、应保持哪些关系，以及哪些知识事实支持这些视觉决策。这类结构化记录也便于人与智能体交互，例如在绘制科普海报时，用户可以明确检查知识内容是否有误。

### 3.3 可执行画布层

在完成绘画内容的构思后，智能体以类似人类绘制草稿的方式来构建草图内容。它根据任务类型选择合适的编程后端，并把对象、文本、空间关系与约束编译成可执行代码。执行后，代码会生成一个类似草图的中间图像，承载图像的核心布局、文字内容和结构关系。

对于复杂构图场景生成，例如强调数量与空间关系的任务，智能体可使用 **SVG** 作为画布后端。SVG 可以把每个对象显式表示为一个节点，并通过坐标和尺寸控制布局。例如，当场景要求生成固定数量的对象、严格左右关系或遮挡关系时，智能体可以直接在 SVG 中创建对应数量的对象节点，而不是依赖图像模型去理解自然语言里“多少个”“位于中心”等含糊描述。

对于文字密集型任务，智能体可以使用 **HTML/CSS** 构建画布。菜单、课程表、网页卡片、说明页、信息图等常包含大量中英文、价格、标题和分区信息。此时，智能体可先用渲染器确定性地绘制文本内容，再将其交给后续图像模型做视觉增强，从而避免让图像模型直接以像素纹理的方式“猜着写字”。此外，在某些无需再次对文字做生成重绘的任务中，也可以采用“文字由代码直接渲染，背景图由图像模型生成”的策略。

对于涉及物理规律的任务，智能体可以调用 **Python 绘图、Canvas 或轻量级 2D/3D 代码** 来构建几何参考。这类任务的关键不在于视觉风格，而在于关系正确。例如基于 Three.js，系统可以显式放置镜面，并渲染小球的反射结果；此时代码就相当于物理模拟器，在最终图像生成前先确定性地建模镜面反射规律。

对于编辑任务，智能体倾向于先构建图像的**分层表示**。它先基于 VLM 理解图像内容，再将各层对象拆分出来，利用 SAM 等工具完成目标分割，并用图像模型补全被遮挡区域。分层表示之后，智能体便可把内容当作可编辑图层，通过 JSONL 格式控制透明度与渲染顺序。例如，用户要求移动某个物体、替换一段文本、或只修改图中某一区域时，智能体可以先定位相应对象层或遮罩区域，再修改其位置、内容或属性。

传统图像生成智能体通常只能通过修改提示词来间接影响图像模型，最终结构仍依赖模型采样结果。相比之下，代码为智能体提供了更自然的操作界面，也与其能力形式高度匹配。智能体更擅长读写结构化文本、调用工具、修改局部代码并检查执行结果，而不是像人类那样通过 GUI 鼠标逐笔绘画。因此，本文并不要求智能体模拟人类 GUI 操作，而是让它通过 CLI/代码直接构建图像结构。对象数量、文本内容、空间位置和图层关系都可以在代码中被清晰表达。

### 3.4 视觉生成与审查层

第三层负责调用现有图像生成或编辑模型完成最终视觉实现，并利用 VLM 或多模态评估能力审查结果。基于第二层得到的可执行代码及其渲染草图，智能体将它们作为视觉条件输入，调用 Qwen-Image、Nano Banana 等具备图像生成与编辑能力的模型完成最终生成。此时，图像模型不再需要从零规划整个场景结构，而只需要在现有草图与结构约束基础上补充纹理、光照、材质与真实感。

在生成场景中，图像模型更像是一个**视觉实现器**：它基于代码草图和文本条件做自然化渲染，把能力集中在纹理、光照、材质、细节与整体真实感上，而不是同时承担复杂规划、计数、文字排版和物理推理等任务。另一方面，相比仅依赖 LLM 生成视觉代码，图像模型也突破了代码表达的上限。代码草图擅长表达结构、布局和文字，但难以承载复杂自然纹理、真实光照和开放场景细节；图像生成模型则可以在保留草图结构的同时，把视觉内容从简单 UI 或示意图扩展到更自然、更真实的场景。

在编辑场景中，智能体基于第二层提供的可编辑对象、分层表示或局部遮罩来执行修改。例如在“移动一个杯子”“替换海报标题”“改变某个物体颜色”这类任务中，智能体可以先确定需要修改的对象层或区域，再调用图像编辑模型完成局部视觉更新。由于编辑目标与范围已由可执行画布事先给出，模型不必重新理解整张图，因此在局部一致性和非目标区域保留上更有优势。

生成之后，智能体的 Review 模块会验证最终输出是否与用户目标和初始结构化记录一致。在传统端到端图像生成中，虽然 VLM 或用户最终也能看出结果有误，但这种黑盒观察难以细粒度定位根因。而在 GenClaw 中，VLM 可以通过检查整个流程中记录下来的中间表示来追踪问题。例如，如果错误源于认知理解偏差或外部知识不准确，智能体可以回溯到第一层核查搜索工具检索到的 URL 内容；如果问题是对象数量错误或文字缺失，系统可以对照中间代码输出与最终图像来精确定位失败点。通过这种分层追踪机制，智能体能够把认知错误、结构错误与视觉错误拆分到不同层进行定向修复。

## 4. 实验

### 4.1 实验设置

作者在多种图像生成与编辑任务上评估了 **GenClaw**。所选基准包括：

- **GenEval++**：评估复杂场景指令遵循能力
- **LongText-Bench**：评估长文本渲染能力
- **ImgEdit**：评估图像编辑能力
- **Mind-Bench**：评估世界知识与推理能力

对比基线包括 GPT-Image、Qwen-Image、Nano-Banana、BAGEL 等当前领先的开源与闭源生成模型；同时，为了明确区分“代码即画笔”与传统“提示词改写型智能体”，作者还纳入了以提示词重写为主的图像智能体系统，如 GenAgent 与 Mind-Brush。

在实现上，GenClaw 的智能体骨干采用 **Claude-Opus-4.6**，默认生成器为 **Gemini-3.1-Flash-Image**。智能体负责把用户意图翻译成结构化记录与可执行画布，而生成器则在草图、文字层、局部遮罩或特定编辑约束条件下完成最终自然渲染。后端代码会根据任务动态切换：复杂构图和分层编辑使用 SVG；海报与长文本任务使用 HTML/CSS 或 SVG 文字层；物理与几何预览则使用 Python、Canvas 或 Three.js。

### 4.2 主要结果

#### 4.2.1 可执行结构提升复杂构图控制力

在 **GenEval++** 上，GenClaw 受益于智能体生成的显式 SVG 布局引导，在 Counting 与 Spatial 等任务上表现出明显优势。论文指出，即便是强大的闭源模型，在仅依赖文本控制时，面对数量和空间关系这类容易被自然语言压缩或错配的任务，仍然存在明显困难。

这一结果也将 GenClaw 与 Mind-Brush、GenAgent、PromptEnhancer 等“提示词重写范式”区别开来。重写式智能体虽然可以通过扩展与重组提示词提升基础模型的指令遵循能力，但其中间状态仍然是自然语言，无法真正锁定物体数量和空间坐标。对于需要精确属性绑定的任务，它们仍然只能不断修改提示词并重新采样。相比之下，GenClaw 在生成前就把对象节点、位置、尺寸和图层写入 SVG 画布，视觉解码器只需在这一结构上补充纹理与真实感。

论文给出的 GenEval++ 关键结果如下（Overall 越高越好）：

- Qwen-Image：0.668
- GPT-Image-1：0.739
- GPT-Image-1.5：0.750
- Gemini-3.1 Flash-Image：0.775
- Mind-Brush：0.782
- **GenClaw：0.878**

作者认为，这一提升并不是因为“提示词更长”，而是因为智能体先基于代码与可执行检查画出了结构化草图。

#### 4.2.2 文本渲染与海报生成

在 **LongText-Bench** 上，GenClaw 在中英文长文本任务中都取得了明显优势。其根本原因在于任务分工的改变：文字不再由图像模型在像素空间“猜写”，而是由 HTML/SVG 文字层进行确定性渲染；图像模型主要负责背景、风格与视觉细节，因此不再需要同时承担字符生成、版式组织与真实渲染。

LongText-Bench 的关键结果如下：

- GPT-Image-1：EN 0.956 / ZH 0.619
- Gemini-3.0 Pro-Image：EN 0.981 / ZH 0.949
- Seedream 4.5：EN 0.989 / ZH 0.987
- **GenClaw：EN 0.989 / ZH 0.988**

论文中的海报案例进一步说明了“文字渲染”与“知识补全”之间的关系。以“2026 世界杯”或《滕王阁序》为例，模型不仅要把文本写对，还要知道应该展示哪些信息。前者涉及实时赛事、主办信息、视觉符号与布局组织；后者涉及古典文本内容、文化背景与排版美学。GenClaw 可以调用搜索工具补齐知识，将其组织为结构化内容，再写入 HTML/SVG 画布；最终图像模型只需负责背景图、装饰元素与整体风格生成。

#### 4.2.3 物理模拟作为可执行视觉推理

论文还探索了 GenClaw 在物理模拟图像生成任务上的能力。与普通构图或文本渲染不同，这类任务更强地考验图像生成模型是否真正理解并呈现复合物理规律。作者指出，无论是镜面反射，还是弹簧、压力、浮力等具体物理场景，直接图像生成模型的表现都不理想，因为图像模型更擅长拟合文本中提到的对象内容，但不一定理解这些元素之间应满足什么物理关系。

相比之下，GenClaw 在执行图像生成之前，会先利用代码构建一个简化物理或几何模型。此时代码不仅是视觉布局工具，也承担中间建模的作用。例如在镜面问题中，系统可先基于 Three.js 放置镜面材质、光源和物体位置，进行确定性渲染，从而为最终图像生成提供反射位置参考。对压力、弹簧等问题，智能体也会先解析物理变量、约束关系和可视化目标，再利用 Python、Canvas 等构造中间模拟图像。

这些中间图像并不追求最终真实感，而是作为一种“物理草稿”或“符号世界模型”，先保证反射、弹簧形变、力方向、液面或几何关系正确，再交给视觉解码器做自然化渲染。作者据此认为，“代码即画笔”的价值不只在于“画得更整齐”，还在于：当生成任务包含可形式化的世界规则时，代码画布能够在像素生成之前先执行一部分世界建模。

#### 4.2.4 ImgEdit 上的图像编辑

在 **ImgEdit** 基准上，作者除了标准 VLM 整体打分之外，还特别关注**未编辑区域的一致性**。因此他们参考 CoCoEdit，利用 mask 标注，仅在未编辑区域上计算 PSNR 与 SSIM。

ImgEdit 关键结果如下：

- FLUX.1 Kontext：VLM 3.43 / PSNR 19.59 / SSIM 0.592
- Qwen-Image-Edit：VLM 3.70 / PSNR 17.64 / SSIM 0.526
- CoCoEdit：VLM 3.79 / PSNR 19.13 / SSIM 0.555
- FLUX.2 Klein 9B：VLM 4.33 / PSNR 22.50 / SSIM 0.695
- Gemini-3.1 Flash：VLM 4.51 / PSNR 20.19 / SSIM 0.583
- GPT-Image-1.5：VLM 4.69 / PSNR 16.36 / SSIM 0.433
- **GenClaw：VLM 4.29 / PSNR 27.87 / SSIM 0.718**

可以看出，虽然某些模型在 VLM 评分上较高，但会对未编辑区域造成较强破坏。GenClaw 在像素级保真指标上明显更优，表明它对非目标区域的扰动更小。作者认为，这一优势直接源于**分层编辑范式**对非目标区域的天然保护作用。

#### 4.2.5 Mind-Bench 上的知识落地

**Mind-Bench** 是一个关注“知识驱动”和“推理驱动”图像生成的基准，用于测试模型在生成前是否正确理解隐含意图与外部事实。实验结果显示，相比基础图像模型，图像智能体整体更有优势，包括 Mind-Brush 与 Nano-Banana-Pro；而 GenClaw 则进一步取得非常强的竞争结果。

Mind-Bench 总体结果如下：

- GPT-Image-1：0.17
- GPT-Image-1.5：0.21
- FLUX 2 Max：0.23
- Nano Banana：0.22
- FLUX 2 Pro：0.22
- Nano Banana Pro：0.38
- Mind-Brush：0.31
- **GenClaw：0.57**

作者指出，相比仅依赖单次 Google 搜索的 Mind-Brush，GenClaw 显著优化了获取外部知识与检索参考图的搜索流程：它采用多轮搜索机制，并能从多个候选中筛选最合适的参考样本。这说明，把理解从生成中解耦，再引入外部知识与显式推理，能够有效提升图像模型在知识密集型任务上的表现。

例如，在某地点街景合成任务中，智能体会先调用推理工具理解输入地图内容，再通过网络搜索获得该地点真实街景作为参考，然后才进行图像生成；在需要确定事实知识的任务中，如精确 NBA 比分，智能体会先用搜索工具锚定具体事实，再进行视觉渲染。这一机制也让用户能够以相对“白盒”的方式审阅结果，更容易追踪整个生成链路中的事实正确性。

## 5. 局限性与未来工作

尽管 GenClaw 展示的代码驱动智能体范式显著增强了空间可控性与构图精度，但作者也指出了若干限制：

### 5.1 对底层生成模型依赖较高

由视觉代码（如 SVG、HTML）渲染出的草图本身是抽象的。要把这些抽象结构转化为高保真、照片级真实感图像，底层图像生成模型需要具备非常强的泛化能力。作者在实验中观察到，当前开源条件生成模型常常难以胜任这一任务，容易产生明显伪影、纹理退化，或只是简单保留原始 SVG 的扁平风格，而无法实现真实化渲染。因此，在当前阶段，要充分验证这一解耦范式，仍需依赖 Gemini-3.1-flash Image 这类强大的前沿模型。

### 5.2 效率开销与收益递减

GenClaw 引入了多步骤智能体流水线，不可避免地带来明显的推理延迟与计算开销。对于需要精确控制的复杂生成任务，这一时间成本是合理的；但对简单直接的任务而言，相比传统一步式端到端生成，这样的长链路会显得冗余且低效。并且随着基础图像生成模型的原生能力不断增强，许多当前仍需智能体工作流处理的复杂任务，未来可能会被模型直接解决，从而削弱智能体架构带来的边际收益。

### 5.3 代码生成稳定性风险

把自然语言翻译为可执行代码，本身就带有不稳定性。LLM 生成代码并非万无一失，可能会产生坐标计算偏差、图层遮挡关系错误、元素缩放比例失衡等问题。这些代码层错误会直接反映到草图渲染结果中，继而导致最终图像的空间布局或细节错位，从而限制系统在某些场景下的稳定性。

## 6. 结论

本文强调，真正的范式变化，并不是从“提示词工程”走向“更复杂的提示词工程”，而是从“让模型一次性猜出整张图”，转向“让智能体像人类画家一样，借助代码一步步搭建图像骨架”。围绕这一理念，作者提出了**代码驱动的智能体图像生成范式**，并在系统 **GenClaw** 中加以实现：通过 **Conceptualize -> Sketch -> Color** 的流程，让 LLM 专注于其擅长的逻辑与结构，把图像生成模型的能力重新聚焦到像素与纹理。

这种解耦设计不仅在复杂构图、文字渲染、物理模拟与分层编辑等任务中展现出更强的可控性，也让整个生成过程变得透明且可追溯：任何失败都可以定位到检索、代码生成或最终渲染中的具体阶段，这是端到端黑盒模型天然不具备的优势。

作者最后指出，正如 Claude Code、Cursor 等代码智能体正在深刻重塑软件工程范式一样，图像生成领域也正经历类似演进，并逐步汇入更广义的智能体浪潮。过去的生成模型主要以被动聊天框形式响应，而未来的视觉生成系统将不可避免地转向更主动、更智能体化的范式。**GenClaw** 代表了这一方向上的初步探索，作者希望这项工作能为构建下一代高可控、可解释、且具备深度推理能力的视觉创作系统提供参考。
